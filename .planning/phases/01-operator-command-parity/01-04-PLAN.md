<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator intent -> destructive write | publish/push/resubmit cause irreversible store/device side effects; the confirm-gate is the boundary |
| CLI -> /api/v1 (POST, Bearer PAT) | The actual write crosses to the backend (which executes on receipt — not preview-gated) |
| confirmGate JSON output -> caller | --json preview emits synthesized verb metadata to stdout |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-13 | Elevation of privilege | confirm-gate on publish/push/resubmit | mitigate | PRIMARY HIGH-SEVERITY THREAT. Without --confirm the verb MUST perform no write and exit 3 (D-04/D-05). Enforced by `const gated = confirmGate(...); if (gated !== null) return gated;` BEFORE any apiFetch. Tests assert requests.length===0 on the no-confirm path for all three verbs — a regression that fires a POST without --confirm fails the suite. |
| T-01-14 | Tampering | --stores token mapping (publish) | mitigate | Aliases apple/google map to canonical apple_appstore/google_playstore; the body always sends canonical AppStore enum tokens. Unknown tokens pass through and are rejected server-side (422), surfaced by the Plan 01 catch — the CLI does not silently drop or invent store targets. |
| T-01-15 | Information disclosure | push recipients_count / preview | mitigate | The no-confirm preview omits the recipient count (v1 exposes it only post-send, Pitfall 2) — no pre-send audience-size leak. The count is read from the v1 201 envelope only after an explicit --confirm send. No PAT in the preview or output. |
| T-01-16 | Information disclosure | resubmit credential block output | accept | The CUSTOMER_ASC_CREDENTIAL_MISSING block prints the server-provided message + dashboard_url (a public Apple-connect link) + next_action — all non-secret, server-curated. The PAT is never logged; --json stays verbatim v1 envelope. |
| T-01-17 | Repudiation | destructive write confirmation | mitigate | exit code 3 (distinct from 1) lets CI/Phase 2 `ship` detect a blocked gate vs a real failure (D-07), giving callers an auditable signal that a write was intentionally withheld. |
</threat_model>
