<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI -> /api/v1 (POST build / PATCH configure, Bearer PAT) | Write requests cross to the backend, which enforces ownership/state/credentials |
| Operator flags -> request body | User-supplied platform/branch/config values become the request body |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Tampering | build branch/platform inputs | accept | The CLI passes platform/branch through; the v1 Build/StoreRequest validates them server-side (Rule::in + regex /^[A-Za-z0-9._\/-]+$/). The CLI must NOT re-implement validation (API is the authority); invalid input returns a 422 surfaced by the Plan 01 catch. |
| T-01-10 | Elevation of privilege | build prerequisite gate (APP_BLOCKED, SUBSCRIPTION_INACTIVE, APPLE_*) | mitigate | The CLI does not bypass prerequisites — they are enforced server-side and surfaced as actionable blocked states via the Plan 01 catch (next_action + dashboard_url). The CLI never auto-retries or suppresses a prerequisite_failed. |
| T-01-11 | Tampering | configure injected_css / injected_javascript | accept | These fields inject content into the user's OWN app webview; the backend owns sanitization/scoping. The CLI is a pass-through editor of the user's own resource (ownership enforced by ability:user + 404-on-not-owned). Out of CLI scope to sanitize. |
| T-01-12 | Repudiation | build/configure are non-confirm writes | accept | build (reversible — produces a build artifact, no store publication) and configure (editable) are intentionally NOT confirm-gated (only publish/push/resubmit are, D-04). Reversibility is the rationale; the destructive trio is gated in Plan 04. |
</threat_model>
