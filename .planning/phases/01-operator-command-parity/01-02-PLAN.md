<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI -> /api/v1 (GET, Bearer PAT) | Read verbs send the PAT to fetch curated resources |
| CLI stdout -> operator/agent | --json emits the raw v1 body to the caller |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Information disclosure | --json passthrough of v1 body | mitigate | --json prints only the parsed v1 response body verbatim (D-08). The body is the server's curated allowlist (AppResource / two-field rejection / recipe collection) — it never contains the PAT or config. No CLI-side field injection. |
| T-01-06 | Information disclosure | status --build raw EAS/GitHub fields | accept | AppBuildResource exposes eas_build_id/eas_build_url/github_run_url/artifact_url. These are present on the v1 user-PAT surface by backend design (not secret on this surface, RESEARCH note). The CLI mirrors v1 verbatim — no drift. |
| T-01-07 | Information disclosure | 404 state-probe guard | accept | The backend returns 404 (not empty 200) for non-REJECTED apps to prevent state probing; the CLI surfaces "No active rejection" without revealing internal state. The guard is enforced server-side; the CLI only renders. |
| T-01-08 | Spoofing | reused apiFetch Bearer auth | mitigate | Read verbs route through the existing apiFetch (Authorization: Bearer <PAT>); no new auth code, no weakening of the 401 -> `appo login` path. |
</threat_model>
