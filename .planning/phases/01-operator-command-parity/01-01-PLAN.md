<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI process -> ~/.appo/config.json | The CLI reads the full-lifecycle user PAT (0600, owner-only) for Bearer auth |
| CLI -> /api/v1 (network) | Bearer PAT crosses to the backend; backend is the authority |
| Test process -> ~/.appo/config.json | The fetch-stub helper writes/reads a test token in the real config path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information disclosure | confirmGate `--json` output | mitigate | Preview object contains only verb metadata (will/app_id/stores/title) synthesized client-side — never the PAT or config. printPreview/confirm JSON emit only the passed `preview` keys; no token reference exists in this code path. |
| T-01-02 | Tampering | Test helper writes ~/.appo/config.json | mitigate | `resetMockFetch()`/teardown MUST restore or clear the config so the real user token is not left altered or leaked into test artifacts. stubToken writes only a dummy 'test-pat', never a real credential. |
| T-01-03 | Elevation of privilege | exit-code-3 gate (confirmGate) | mitigate | The gate is the primary defense against accidental destructive writes (publish/push/resubmit). confirmGate returns 3 and the caller MUST `return gated` before any POST — Plan 04 verbs depend on this contract; foundation tests assert no fetch fires when gated. |
| T-01-04 | Information disclosure | extended error catch | accept | The catch prints `err.message` + envelope `code`/`details.{next_action,dashboard_url}` — all server-curated, non-secret fields (the v1 envelope never contains the PAT). dashboard_url is a public Apple-connect link. No token is ever logged. |
</threat_model>
