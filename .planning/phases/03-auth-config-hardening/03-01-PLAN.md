<objective>
Restructure `src/config.mjs` from the flat `{ token, api_base }` shape into a profile-aware
gateway `{ current, profiles: { <name>: { api_base, token } } }`, with read-time normalization of
the legacy flat config (existing users are never logged out), env-var precedence for `APPO_TOKEN`
and active-profile selection, and a test-isolation seam (`APPO_CONFIG_HOME`) so config tests never
touch the real `~/.appo/config.json`.

Purpose: This is the load-bearing foundation for CLI-02 (multi-environment profiles, no clobbering)
and CLI-07 (`APPO_TOKEN` ephemeral precedence). Every downstream auth-surface edit (api.mjs token
source, login --token, logout revoke, whoami, env list/use) consumes the new exports. Delete the
flat-shape readers entirely — no dual definitions, no versioned migration code (CLAUDE.md).

Output: a profile-aware `src/config.mjs` and a new `test/config-profiles.test.mjs` unit suite.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| process env → config store | `APPO_TOKEN`/`APPO_ENV`/`APPO_API_BASE` are read at the `config.mjs` seam; `APPO_TOKEN` is a secret that must never cross into the on-disk file |
| in-memory config ↔ `~/.appo/config.json` | the only persistence boundary; owner-only (0700/0600) is the confidentiality control for stored PATs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information disclosure | `storedToken` / write paths | mitigate | `APPO_TOKEN` is read-only in `storedToken`; no writer sources from it. Unit test asserts the on-disk file never contains the `APPO_TOKEN` value (Pitfall 2). HIGH — a persisted CI secret in a dev home dir is a confidentiality breach. |
| T-03-02 | Information disclosure | `~/.appo/config.json` permissions | mitigate | All writers route through `writeConfig` (mkdir 0o700 / chmod 0o600), reused verbatim — owner-only stays the storage model. No new write path bypasses it. |
| T-03-03 | Tampering / Repudiation | legacy normalization | mitigate | Read-time fold preserves the existing token (no silent logout); unit test asserts a flat config yields the old token, not null — prevents a normalization bug from destroying a user's credential. |
| T-03-04 | Information disclosure | `env list` data shape | mitigate | Profile objects carry tokens; this plan keeps tokens out of any render path (the CLI render lives in Plan 02). Data-shape test confirms tokens live only in the profile object, consumed by render code that must elide them. |
| T-03-05 | Elevation of privilege | `APPO_CONFIG_HOME` test seam | accept | The env override only relocates the config path; it grants no new capability (the file is still 0600). Useful as a documented CI escape hatch. Low risk — same trust as choosing `$HOME`. |
</threat_model>
