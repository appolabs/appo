<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI → `/api/v1` (Bearer) | apiFetch crosses untrusted network with the PAT in the Authorization header; the 401 response is attacker-influenceable text |
| pasted `--token` PAT → config store | `loginWithToken` decides whether an unvalidated, user-supplied secret is persisted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Information disclosure | apiFetch 401 message | mitigate | The env-named 401 string interpolates ONLY the env name, never any token substring (D-13). Test asserts the message contains the env and not the stub token. HIGH — a token in an error printed to logs/CI is a leak. |
| T-03-07 | Information disclosure | `APPO_TOKEN` on the wire | mitigate | `APPO_TOKEN` is consumed via `storedToken(env)` and sent as `Authorization: Bearer`; it is never persisted (Plan 01 invariant, re-asserted here on the config bytes). |
| T-03-08 | Spoofing / weak auth | `loginWithToken` validation | mitigate | A pasted PAT is validated with `GET /api/v1/apps` BEFORE any write; a 401 refuses and stores nothing (Pitfall 6) — prevents persisting a typo'd/forged token that would 401 every later call. |
| T-03-09 | Information disclosure | `--token` echo | mitigate | `loginWithToken` never logs/echoes the `pat`; refusal message names the api_base, not the token. Test asserts no captured output contains the pasted PAT. HIGH. |
| T-03-10 | Tampering | device-flow write | mitigate | Device login writes via `writeProfile(env, ...)` (merge, no clobber) — logging into one env cannot overwrite another's stored credential. |
</threat_model>
