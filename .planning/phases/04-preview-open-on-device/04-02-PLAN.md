<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI -> Appo `/api/v1` | Authenticated GET with a Bearer PAT; backend owns access control |
| API response -> terminal/stdout | Untrusted server payload rendered to the user; `--json` is machine-consumed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Information disclosure | PAT in `--json` / logs | mitigate | The verb reuses `apiFetch`, which sets the PAT only in the Authorization header and never logs it; `--json` prints ONLY the response body (`JSON.stringify(res)`), never the request/token. Test asserts `--json` output equals the stubbed body exactly (no extra fields) |
| T-04-05 | Information disclosure | resource-existence leak | accept | Backend returns 404 (not 403) for non-owned apps [VERIFIED: V1PreviewController::resolveOwnedApp]; the CLI renders the 404 generically via renderError — no existence inference added client-side |
| T-04-06 | Spoofing / Tampering | preview token in preview_url/deeplink | accept | The preview_url/deeplink CONTAIN a preview token by design (the open-on-device mechanism) — minted server-side, intended to be shown/QR-encoded; not the auth PAT [VERIFIED: api.mjs never logs the PAT] |
| T-04-07 | Denial of Service | over-long preview_url -> wide QR | accept | ECC level M keeps a route+token URL at ~v2-v4 (<=45 cols); the QR unit test (Plan 01) bounds width <= 80. A pathologically long URL still renders, just wider — no crash |
</threat_model>
