<objective>
Vendor a dependency-free terminal QR encoder as `src/qr.mjs` and expose a pure
`renderQr(text) -> string` half-block renderer, with pure-function unit tests proving
the output is structurally scannable (correct finder patterns, timing, quiet zone).

Purpose: The scannable QR is the load-bearing, novel work of Phase 4 — a nearly-correct
hand-rolled encoder is worse than none (it renders but won't scan). This plan isolates that
risk in a self-contained, fully unit-testable module with NO dependency on the CLI verb, so it
runs in Wave 1 in parallel with nothing blocking it and is ready for the verb in Wave 2.

Output: `src/qr.mjs` (vendored Nayuki qrcodegen, MIT, converted TS->ESM, + `renderQr`),
`test/unit/qr.test.mjs`. RUNTIME `dependencies` stay empty — the encoder is OUR source.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | vendored `src/qr.mjs` | mitigate | Pin a specific upstream commit and record the hash in the file header; vendor the source (no npm install) so the shipped bytes are reviewable in-repo and cannot drift via a transitive update |
| T-04-02 | Elevation of Privilege | vendored encoder | accept | Encoder is pure/offline: no network, no `eval`, no `fs` — a string->matrix transform. Verify by inspection during conversion that no I/O or dynamic eval is introduced |
| T-04-03 | Information disclosure | `renderQr` output | accept | The encoder only encodes what it is given; the URL it will encode (preview_url) is a public open-on-device target by design, not a secret. The auth PAT is never passed to this module |
</threat_model>
